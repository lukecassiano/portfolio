## Spacing Scale

Identical to Phase 2 (inherited — do not re-specify in implementation):

| Token | Value | Phase 3 Usage |
|-------|-------|---------------|
| xs | 4px | Inline gaps (stat label to value, caption gap) |
| sm | 8px | Back-link arrow-to-text gap |
| md | 16px | Paragraph gap; stack gap between heading and opening sentence |
| lg | 24px | Wordmark → tagline → message stack on stub pages |
| xl | 32px | Heading → body copy gap within a case study section; horizontal page padding mobile (`px-8`) |
| 2xl | 48px | Between case study sections (top padding) |
| 3xl | 64px | Between case study sections (bottom padding); page horizontal padding desktop (`px-16`) |

**Section rhythm on /sandbar:** Each `<section>` receives `py-16` (64px) vertical padding. The gap between the preceding section's last paragraph and the next section's heading is effectively `64px + 64px = 128px` — this is deliberate editorial white space, not an error.

**Prose column:** `max-w-2xl` (Tailwind default: 42rem / ~672px) centered with `mx-auto`. Applied to ALL content including headings — headings do NOT break out wider. No exceptions.

Source: REQUIREMENTS.md SAND-02 (max-w-2xl constraint), Phase 2 UI-SPEC (spacing scale inherited).

## Typography

**No new tokens.** Phase 3 uses the exact 4-token scale from Phase 2. Adding a fifth token is not justified.

### Type Scale (inherited from Phase 2 — authoritative)

| Token | Font | Size | Weight | Line Height | Phase 3 Usage |
|-------|------|------|--------|-------------|----------------|
| `display` | Fraunces `.wordmark` | `clamp(56px, 10vw, 120px)` | 800 italic | 1.0 | NOT used in Phase 3 — reserved for "Luke Cassiano" intro only |
| `wordmark` | Fraunces `.wordmark` | `clamp(36px, 6vw, 72px)` | 800 italic | 1.1 | /sandbar h1 page title, all h2 section headings, hero stat "87%", stub page wordmarks |
| `body` | IBM Plex Mono | 16px | 500 | 1.5 | All prose paragraphs on /sandbar; stub honest message text |
| `label` | IBM Plex Mono | 12px | 500 | 1.4 | Stat labels, captions, back navigation link, pipeline diagram caption, metadata |

### Section Heading Assignment

All five case study section headings (Problem, Insight, Product, Under the Hood, What's Next) use the `wordmark` token: Fraunces `.wordmark` at `clamp(36px, 6vw, 72px)`, weight 800 italic, line-height 1.1.

These are `<h2>` elements. The page `<h1>` is "Sandbar" (also `wordmark` token).

Rationale: The `display` token is too large for prose section headings — it is used only for "Luke Cassiano" on the intro scene. Using `wordmark` for both the h1 and h2 elements in this editorial context is correct: the h1 is the wordmark/brand identity; the h2s are section markers within a long-scroll article. Both use the same type size but carry different semantic weight via heading level.

Source: Phase 2 UI-SPEC (4-token constraint), REQUIREMENTS.md SAND-04.

### Prose Typography Rules

- All prose paragraphs: IBM Plex Mono 500, 16px, line-height 1.5, max-w-2xl, ink `#1A1A1A` on cream `#F5F3EE`
- No serifs in body text. IBM Plex Mono is the ONLY font for paragraphs, captions, labels, metadata
- Fraunces is ONLY used for: page h1, section h2 headings, hero stat numerals, stub wordmarks
- No italic for body text. Italic belongs to Fraunces only (via `.wordmark` class)
- Letter-spacing on labels: `tracking-[0.08em]` uppercase, same as Phase 2 stat labels

### Tailwind Class Reference (inherited)

| Token | Tailwind classes |
|-------|-----------------|
| `wordmark` | `font-serif italic font-extrabold` + inline style `fontSize: 'clamp(36px, 6vw, 72px)'` + `lineHeight: 1.1` |
| `body` | `font-mono font-medium text-base leading-relaxed` (16px / 1.5) |
| `label` | `font-mono font-medium text-xs` (12px) |

Source: Phase 2 UI-SPEC typography contract.

## Interaction Contract

### Back Navigation (all three pages)

Style and behavior identical across /sandbar, /belief-agent, /whitehelmet:

```tsx
<a
  href="/"
  className="font-mono font-medium text-xs tracking-[0.08em] uppercase
             hover:opacity-60 transition-opacity duration-200"
>
  ← Back
</a>
```

- Position on /sandbar: inline at top of prose column, `mb-16` below it before the h1.
- Position on stubs: `absolute top-8 left-8 lg:top-12 lg:left-12` — overlaid on the scene, above blob z-index.
- Text color: inherits from page (ink on cream pages; cream on Belief Agent dark page).
- Arrow: Unicode `←` (U+2190), not an SVG icon, not a component. Same pattern as Phase 2 `→` CTAs.
- No underline at rest, no underline on hover. Opacity transition only: `hover:opacity-60 transition-opacity duration-200`.

Source: Phase 2 UI-SPEC CTA link style (same interaction pattern applied to back navigation).

### SceneFadeIn on Stub Pages

Both stub pages use `SceneFadeIn` (same component built in Phase 2) wrapping the content block. Same animation parameters as Phase 2:

| Property | Value |
|----------|-------|
| `initial` | `{ opacity: 0, y: 12 }` |
| `animate` (in view) | `{ opacity: 1, y: 0 }` |
| `transition.duration` | `0.6` |
| `transition.ease` | `'easeOut'` |
| Reduced-motion | `initial` and `animate` both `{ opacity: 1, y: 0 }` |

Source: Phase 2 UI-SPEC fade-in contract.

### /sandbar Case Study — No Scroll Animation

The case study page does NOT use SceneFadeIn or per-section fade-in animations. Rationale: the landing page uses fade-in for impact on first view; the case study is a reading document where fade-in animations would interrupt reading flow and add complexity. Content is immediately visible on load.

Exception: if the executor wants to add a single fade-in on initial page load (just the h1 block), that is permissible using the same SceneFadeIn parameters. It is not required by this contract.

Source: PROJECT.md (minimal motion constraint), REQUIREMENTS.md (no animation requirement for SAND-01–06).

### Pipeline Diagram — /sandbar (SAND-03)

The pipeline diagram is an inline SVG React component, NOT an `<img>` tag and NOT imported via SVGR.

Requirements:
- Component file: `components/sandbar/PipelineDiagram.tsx`
- Must have `aria-label` prop on the `<svg>` element
- Must have `<title>` as first child of `<svg>` (for screen reader compatibility)
- `aria-labelledby` pointing to the `<title>` element id
- Max-width: stays within `max-w-2xl` prose column — no breakout. `width="100%"` on the SVG element.
- SVG is responsive: `viewBox` set, no fixed pixel width/height on the svg element itself.
- Tailwind can be used for the wrapping `div`; stroke/fill values are inline SVG attributes.
- Diagram renders in the "Product" section of the case study.
- A `<figcaption>` below the diagram uses the `label` token: IBM Plex Mono 12px weight 500, ink, `tracking-[0.08em]`.

```tsx
// Structural pattern — executor fills content
<figure className="my-8">
  <svg
    viewBox="0 0 800 200"
    width="100%"
    aria-labelledby="pipeline-title"
    role="img"
  >
    <title id="pipeline-title">Sandbar data pipeline: buoy → model → forecast</title>
    {/* diagram paths, rects, text */}
  </svg>
  <figcaption className="font-mono font-medium text-xs tracking-[0.08em] mt-4 text-center">
    BUOY DATA → NOAA GRID → ENSEMBLE MODEL → SURF FORECAST
  </figcaption>
</figure>
```

Source: REQUIREMENTS.md SAND-03, RESEARCH.md (SVG inline pattern), SUMMARY.md.
